# 콘텐츠 검색 — 설계와 개선 로드맵

문서 본문 검색(E3) 기능의 현재 구조, 성능 수치, 앞으로의 업그레이드 경로를 정리한다.
의사결정은 gstack `/autoplan` 리뷰(2026-06-24)에서 나왔다.

## 현재 구조 — A+ (구현 완료)

"본문 어딘가 있던 문구를 기억해서 찾기"가 핵심 작업이라, 의미검색이 아니라 **문자열 매칭**이 정답.

1. **인덱스 생성** — `gdoc analyze` (CLI)
   - 각 문서 HTML을 Storage에서 받아 `extractText()`로 평문 추출.
   - 임베딩용(4,000자)과 **검색용(50,000자, `SEARCH_MAX`)을 분리** — 긴 문서도 끝까지 검색.
   - `{ docId: 본문텍스트 }`를 `graph/search-index.json`(private 버킷)에 저장.
   - **증분**: 바뀐 문서만 재추출(`mergeSearchIndex`). 상한을 바꾸면 `gdoc analyze --rebuild`로 전체 재생성.
2. **로드** — `useSearchIndex` 훅: 소유자 로그인 시 서명 URL로 1회 fetch, 메모리 상주. 없으면 제목 검색으로 graceful fallback.
3. **검색** — `App.tsx`: 입력어를 디바운스 후, 각 문서에 `searchScore(title, text, query)`로 점수 매겨 매치만 남기고 관련도순 정렬. 본문 매치엔 `contentSnippet`으로 스니펫 표시.
   - 랭킹: 제목 매치(>1000) > 본문 출현 횟수 > 첫 매치 위치.

순수 로직(`shared/searchSnippet.ts`, `shared/searchIndex.ts`)은 단위 테스트로 보호.

### 성능 (실측, 37개 문서)
| 지표 | 값 |
|---|---|
| 인덱스 크기 | 305 KB (문서당 ~8KB, 50k 상한) |
| 최장 문서 색인 | 15,893자 (전체) |
| 키스트로크당 스캔 | ~0.34 ms |
| 최초 로드 | ~50–150 ms (1회) |

### 규모별 추정
| 문서 수 | 인덱스 | 최초 로드 | 스캔/키 | 체감 |
|---|---|---|---|---|
| ~100 | ~0.8 MB | ~0.2s | ~1ms | 무감각 |
| ~500 | ~4 MB | ~0.5–1s | ~5ms | 괜찮음 |
| ~1,000 | ~8 MB | ~1–3s | ~10ms | 로드만 부담 |
| ~5,000 | ~40 MB | 느림 | ~45ms | 재설계 신호 |

스캔은 병목이 아니다. **진짜 비용은 "인덱스 전체를 매 세션 1회 다운로드 + 메모리 상주"** 이며 문서 수에 비례한다.

### 한계
- 전체 인덱스 다운로드(수천 개부터 부담).
- substring 매칭만 — 오타 허용·형태소·의미검색 없음.
- 랭킹은 단순 휴리스틱(출현 횟수 기반).

## B′ — Postgres `ILIKE` 서버 검색 (다음 업그레이드 후보)

**언제**: 문서가 ~1,000개를 넘거나, 인덱스 다운로드 용량이 거슬릴 때.

**설계**
- 추출 본문을 `documents`에 텍스트 컬럼으로 저장(예: `content_text`), 업로드 시 채움.
- 뷰어가 Supabase 클라이언트로 `select ... where content_text ilike '%쿼리%'` (RLS로 권한 자동 적용).
- 스니펫: 반환된 `content_text`에서 클라이언트가 `contentSnippet`으로 생성, 또는 Postgres에서.
- 한글 substring은 `ILIKE`로 토크나이저 없이 동작. 규모가 커지면 `pg_bigm`(또는 `pg_trgm`) GIN 인덱스로 가속.

**장점**: 전체 다운로드 제거(매치만 수신), 수천~수만 건 확장, 메모리 부담 없음, RLS 그대로.
**비용**: 마이그레이션(컬럼 추가 + 백필) + 업로드 경로 변경 + 검색마다 네트워크 왕복(디바운스 필수) + (대규모 시) `pg_bigm` 확장 가용성 확인.

**마이그레이션 스케치**
1. `alter table documents add column content_text text;` + (선택) `create extension pg_bigm; create index ... using gin (content_text gin_bigm_ops);`
2. 업로드(`cli/upload.ts`)에서 `extractText` 결과를 `content_text`에 upsert.
3. 기존 문서 백필(1회 스크립트).
4. 뷰어: `useSearchIndex`(JSON 로드)를 ILIKE 질의로 교체, 디바운스 유지.
5. `search-index.json` 생성 로직(`analyze`)은 제거하거나 폴백으로 유지.

## D — 시맨틱 검색 (선택, 보완 기능)

이미 `analyze`가 문서 임베딩(MiniLM)을 만든다. 쿼리를 임베딩해 코사인 랭킹하면 "비슷한 주제/의미"로 찾기가 된다.
- substring("이 문구 들어간 문서")과 **다른 작업**이라 대체가 아니라 보완.
- 쿼리 임베딩 실행처(브라우저 모델 ~90MB or 서버 엔드포인트)가 필요.
- 문서 단위 벡터라 문서 레벨 관련도까지만(문구 위치는 못 짚음).

## 결정 로그
- **2026-06-24 — A+ 채택.** 1인 도구·수십~수백 문서 규모에서 substring이 핵심 작업의 정답이고, 유일한 실질 약점(4,000자 컷·랭킹 부재)을 인프라 0으로 제거. B′의 서버 왕복·마이그레이션은 이 규모에 과설계. 다운로드/규모가 문제될 때 B′로 승격, 의미검색은 D로 별도 추가.
